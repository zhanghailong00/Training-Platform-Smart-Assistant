# 实训平台智能助手 — 实习项目总结

> 适用于面试中的项目介绍

---

## 项目背景

实训平台已有课程和学生考试功能，老师需要在平台上设计考试内容。传统做法是老师手动出题，效率低、工作量大。目标是做一个**试题生成助手**，老师上传课件文档或输入教学内容，AI 自动生成结构化试题 JSON，直接对接平台题库。

---

## 我的工作

### 整体架构设计

从零搭建了一个模块化的 LLM 后端服务，采用 7 层分层架构：

```
core → schemas → utils → components → services → api → ui
```

每层职责清晰，只依赖下层，不跨层调用。核心业务逻辑只有一个编排入口（`exam_service.py`），同时被 Gradio UI 和 REST API 调用，避免双入口行为不一致。

**技术栈**：Python + FastAPI + Gradio + DeepSeek API + PyMuPDF + Pydantic

### 数据模型先行

在写任何业务逻辑之前，先对接同事的题库 JSON schema，用 Pydantic 严格建模。定义了 4 个数据模型文件（枚举、请求、试题、响应），确保生成结果格式与平台题库完全对齐，减少联调返工。

### 7 层 JSON 稳定性保障体系

大模型输出 JSON 天然不稳定，构建了一套完整的保障流水线：

| 层 | 方案 | 解决什么问题 |
|---|---|---|
| ① 模型约束 | DeepSeek JSON Mode | 保证语法合法，杜绝 markdown/解释文本 |
| ② 文本修复 | 4 层正则提取 + 格式修复 | 去除代码块标记、修复尾随逗号 |
| ③ Schema 校验 | Pydantic 严格校验 | 保证字段名、类型、必填项正确 |
| ④ 业务规则校验 | 自定义 BusinessValidator | 保证单选题 1 个答案、多选题至少 2 个等 |
| ⑤ 自修复重试 | 把校验错误反馈给 LLM 修正，最多 3 次 | 不是盲重试，是针对性修正 |
| ⑥ 降级返回 | 部分题型成功时不阻塞 | 一个题型失败不会全盘丢弃 |
| ⑦ 人工兜底 | 同事入库时审核 | 最终防线 |

这套体系从"依赖模型输出"转向"通过工程手段保证输出质量"，把 JSON 可用性从"大部分时候能用"提升到"每次都能用"。

### 文档解析

支持 PDF 和 Word 格式的课件文档上传解析，提取文本内容作为 LLM 的输入素材。

### 双入口设计

- **Gradio UI**：老师直接浏览器打开使用，上传文档、选择题型、预览结果
- **REST API**：供实训平台后端调用，返回 JSON

同一个编排逻辑同时服务两个入口。

### Prompt 模板独立管理

Prompt 不硬编码在代码中，而是放在 YAML 配置文件中按题型拆分。非开发人员也可以直接修改 Prompt 调优，不改代码。

### 多模型自动切换（Fallback）

接入两个不同厂商的 LLM（DeepSeek + Qwen），主模型不可用时自动切换到备用模型，对业务代码完全无感。

切换机制：
- 任何错误都会触发切换：超时、API 不可用、欠费、Key 失效
- 每个模型内部有 2 次指数退避重试
- 两个厂商独立账户，一个欠费不影响另一个
- 配置级改动，只需在 `.env` 中增加 fallback 配置项，不改代码

这保证了服务的可用性——不会因为某个模型的服务波动或账户余额问题导致整个系统不可用。
40s减少到25s

  师用手机拍下纸质试卷的照片，或者直接上传一张试题图片到平台。这张图片被传到后端后，系统先调用 qwen3.5-ocr 视觉模型识别图片中的文字——这个模型是专门做 OCR  的，能把图片里的文字原样提取出来。比如图片里有一道题"计算机科学的奠基人是__B__，A、查尔斯·巴贝奇 B、图灵 C、阿塔诺索夫 D、冯·诺依曼"，OCR 模型就会把这行文字识别出来。
  但这只是原始文字，还不是结构化的试题数据。接下来，系统把这串文字作为 Prompt 的一部分，发给大模型（Qwen），让大模型去理解这段文字，从中提取出题型、题干、选项、正确答案，然后输出结构化的
  JSON。这个环节叫"结构化提取"，用的是 prompts/ocr_extract.yaml 里专门设计的 Prompt，告诉大模型"你要把 OCR 文字转成 JSON 格式的试题"。

  大模型返回 JSON 后，系统做一次 Pydantic 校验，确保字段名、字段类型都正确。校验通过后，把结果组装成和试题生成接口完全一样的 subjectBanks 格式，返回给调用方。如果是缺失的字段（比如
  level、levelName），系统会自动补上默认值。

  输出的 JSON 直接给 Java 后端，Java 后端把它入库到平台题库，整个过程就完成了。

  关键点：OCR 识别出来的文字不是直接入库，而是先经过 LLM 结构化提取，把非结构化的文字转成结构化的试题 JSON，再入库。这和试题生成的区别在于——试题生成是"根据教学内容出新的题"，OCR
  是"把纸质试卷上的题转成电子格式"。

## 面试中可能被问到的问题

### Q：为什么选择 DeepSeek 而不是 GPT？
性价比高，API 兼容 OpenAI 格式，后续切换模型只需改一行配置。

### Q：JSON 校验失败为什么还要重试 3 次，而不是直接报错？
因为 LLM 输出的随机性，第二次重试往往就能输出正确的 JSON。而且不是简单重试——把错误原因反馈给模型，让它有针对性地修正，成功率远高于盲重试。

### Q：Prompt 为什么要用 YAML 管理？
因为 Prompt 是核心资产，需要频繁调优。放在 YAML 里可以热加载、按题型拆分、非开发人员也能改，而不是硬编码在 Python 文件里。

### Q：为什么没有用 LangChain 这类框架？
项目规模小，引入框架反而增加复杂度。自定义的分层架构更轻量、可控、容易 debug。

### Q：Gradio 和 FastAPI 怎么共存的？
Gradio 6.x 支持 `gr.mount_gradio_app()`，把 Gradio 应用挂载到 FastAPI 的 `/` 路径下，API 端点在 `/api/v1/*`，同一个 `uvicorn` 进程跑起来。
